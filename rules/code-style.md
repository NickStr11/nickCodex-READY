# Code Style Rules

## Базовый принцип

- Сначала смотри на стиль текущего репозитория и продолжай его.
- Если явного стиля нет, выбирай простой, читаемый и предсказуемый код.
- Специфика проекта важнее общего style guide.

## Как должен выглядеть код

- Имена должны объяснять смысл без лишних комментариев.
- Функции и методы держать небольшими, если это не вредит читабельности.
- Один блок кода = одна понятная ответственность.
- Не тащить абстракции на будущее, если сейчас хватает прямого решения.
- Не плодить helper-методы, если они используются один раз и только разрывают чтение.
- Комментарии писать редко и только там, где нужно объяснить почему, а не что происходит.
- Ошибки обрабатывать явно. Не прятать сбои молча.
- Магические числа, строки и флаги выносить в именованные константы, если они не очевидны из контекста.

## При изменениях

- Предпочитать минимальный рабочий diff вместо широкого рефактора без запроса.
- Если меняется поведение, обновить или добавить тесты, если в проекте есть тестовая среда.
- После правок прогонять форматтер, линтер и тесты, если они существуют в репозитории.
- Если форматтер уже принят проектом, не спорить с ним руками.

## Python

- Опираться на читаемость прежде всего.
- Следовать локальному стилю проекта раньше, чем общим правилам PEP 8.
- Имена: `snake_case` для функций и переменных, `PascalCase` для классов, `UPPER_SNAKE_CASE` для констант.
- Предпочитать явные конструкции хитрым однострочникам.
- Не лепить огромные функции и вложенность, если можно рано выйти или вынести понятный шаг.
- Докстринги писать там, где без них неясен контракт или побочный эффект, а не на каждую мелочь.

## TypeScript / JavaScript

- В TS сначала пытаться выразить инварианты типами, а не комментариями.
- Имена: `PascalCase` для типов, интерфейсов, enum values; `camelCase` для функций, свойств и локальных переменных.
- Не использовать префикс `I` для интерфейсов, если проект сам этого не требует.
- Предпочитать `undefined`, а не `null`, если локальный стиль не говорит обратное.
- Предпочитать arrow functions вместо безымянных `function` там, где это не ухудшает читаемость.
- Не экспортировать всё подряд; наружу выносить только реально разделяемые вещи.
- Не использовать `any`, если можно выразить тип точнее без цирка.

## React

- Компоненты и хуки должны быть чистыми: одинаковый ввод -> одинаковый результат.
- Побочки не делать в render; держать их в event handlers, effects или framework-специфичных местах.
- `props` и `state` не мутировать напрямую.
- Хуки вызывать только на верхнем уровне React-функций.
- Компоненты резать по ответственности, а не по количеству JSX-строк.
- Локальный UI-state держать ближе к месту использования; не поднимать его без причины.
- Если логика компонента разрастается, сначала вынести понятный custom hook или чистую функцию, а не городить ещё один слой абстракции ради вида.

## Границы

### Always

- Сохранять локальные соглашения по стилю и архитектуре.
- Проверять затронутые файлы на консистентность после правок.
- Предпочитать явность хитрой компактности.

### Ask First

- Подключение новых зависимостей.
- Массовый рефактор на много файлов.
- Изменение публичных API, схем данных или форматов файлов.
- Ручная правка generated/vendor файлов, если у них есть источник генерации.

### Never

- Не смешивать в одном изменении поведенческий фикс и декоративную уборку без причины.
- Не оставлять мёртвый код, временные заглушки и закомментированные куски на потом.
- Не придумывать стиль поверх уже существующего стиля проекта.
