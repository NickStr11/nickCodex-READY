# PharmOrder / SCLIT_v3: что помнить

Дата: 2026-03-07

## Главное правило
Не лезть в архитектурный рефакторинг вслепую.
Сначала карта зависимостей и контрактов, потому что система сцеплена не только кодом, но и внешней средой.

## Почему это рискованно
Есть связи между:
- локальным компом
- маминым компом
- VPS
- SKLIT / DBF-файлами
- кассой
- relay
- scan inbox/outbox
- алиасами и ценами
- внешними плагинами/скриптами для дельт по прайсам

Любой рефакторинг без карты связей может сломать:
- импорт прайсов
- экспорт заявок
- синк сканов
- резолв неизвестных товаров
- остатки
- внешние плагины и батники

## Что сначала проверить перед любыми изменениями
- кто пишет и читает `scan_inbox` / `scan_outbox`
- кто использует scan_aliases.json
- кто использует `prices.db`
- кто вызывает `/api/scan-items`, `/api/sync/status`, `/api/scan-import/*`
- какие плагины/скрипты стоят на локальном компе и у мамы
- какие пути, форматы файлов и JSON-контракты они ждут
- какие env-переменные и shared папки являются внешним контрактом

## Что видно по проекту SCLIT_v3
- проект лучше держать как модульный монолит, не распиливать на микросервисы
- основные целевые контексты: catalog, ordering/export, inventory, scan_sync, resolver
- главная проблема не в стеке, а в том, что слишком много ответственности в src db.py и src server.py
- README.md и PROJECT_CONTEXT.md устаревшие/пустые, реальная правда сейчас в DEV_CONTEXT.md и коде
- есть явный security smell: в docs/cashbox-system.md лежали живые-looking ключи

## Что видно по VPS
- Ubuntu 24.04
- публично слушаются `22`, `5000`, `8000`, `8080`, `10050`
- `nginx`/`apache` нет
- `ufw` выключен
- SSH сейчас опасный: `PermitRootLogin yes` + `PasswordAuthentication yes`
- сервисы крутятся от `root`
- `pharmorder` слушает `0.0.0.0:8000`
- `scan-relay` слушает `0.0.0.0:8080`
- `scan-relay` запускается из venv другого проекта: `/opt/apteka-bot/.venv`
- в логах `apteka-bot` светится токен Telegram через URL

## Рекомендуемый порядок дальше
1. Не менять архитектуру, пока не будет dependency map.
2. Собрать карту внешних контрактов: файлы, API, плагины, shared папки, компы.
3. Отдельно сделать hardening VPS: новый админ, SSH keys, запрет root/password, ufw, reverse proxy, закрыть app-порты.
4. Только потом трогать внутренний рефакторинг `PharmOrder`.

## Смысл архитектурных изменений потом
Не "сделать красиво", а:
- удешевить будущие правки
- снизить шанс сломать рабочий сценарий
- убрать хрупкость db.py и server.py
- сделать проект понятным не только по памяти
